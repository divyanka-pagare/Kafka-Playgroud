If a topic has 3 partitions and a consumer group has 5 consumers, how many consumers will actually receive messages, and what will happen to the remaining consumers? 🤔

--> Scenario

Topic:

orders
├── Partition 0
├── Partition 1
└── Partition 2

Consumer Group:

Group A
├── Consumer 1
├── Consumer 2
├── Consumer 3
├── Consumer 4
└── Consumer 5
What Kafka does

Kafka follows this rule:

One partition can be assigned to only one consumer within a consumer group.

So:

Consumer 1 → Partition 0
Consumer 2 → Partition 1
Consumer 3 → Partition 2
Consumer 4 → No Partition
Consumer 5 → No Partition
Will Consumer 4 and 5 "wait"?

Yes, they stay idle because there are no partitions available.

However, the more precise answer is:

They remain part of the consumer group but are not assigned any partitions.

What happens if Consumer 2 crashes?

Kafka triggers Rebalancing.

Before:

Consumer 1 → Partition 0
Consumer 2 → Partition 1
Consumer 3 → Partition 2
Consumer 4 → Idle
Consumer 5 → Idle

After Consumer 2 dies:

Consumer 1 → Partition 0
Consumer 4 → Partition 1
Consumer 3 → Partition 2
Consumer 5 → Idle

Kafka automatically reassigns the unowned partition.

Interview Answer

If a topic has 3 partitions and a consumer group has 5 consumers, 
only 3 consumers will actively consume messages because each partition 
can be assigned to only one consumer within the same consumer group. 
The remaining 2 consumers stay idle. If one active consumer leaves the group, 
Kafka performs a rebalance and assigns its partition to one of the idle consumers.

Q. Suppose:

Topic = 10 Partitions
Consumer Group = 3 Consumers

How many partitions will each consumer get?

-->

For:

10 Partitions
3 Consumers

Kafka tries to distribute partitions as evenly as possible.

Example assignment:

Consumer 1
├── Partition 0
├── Partition 3
├── Partition 6
└── Partition 9

Consumer 2
├── Partition 1
├── Partition 4
└── Partition 7

Consumer 3
├── Partition 2
├── Partition 5
└── Partition 8

So:

Consumer 1 → 4 partitions
Consumer 2 → 3 partitions
Consumer 3 → 3 partitions

Kafka won't always assign them in the exact round-robin pattern you wrote because it depends on the partition assignment strategy, but conceptually:

Partitions are distributed as evenly as possible among consumers.

---------------------------------------------------------------------------------------

Number of consumers > Number of partitions
3 Partitions
5 Consumers

Result:

3 Active Consumers
2 Idle Consumers


------------------------


Number of consumers = Number of partitions
3 Partitions
3 Consumers

Result:

Each consumer gets 1 partition

This is usually the ideal scenario.

--------------------------


Number of consumers < Number of partitions
10 Partitions
3 Consumers

Result:

Each consumer gets multiple partitions

Exactly as you described.

Key Rule
Inside the SAME Consumer Group
Group A
├── Consumer 1
├── Consumer 2
└── Consumer 3

Messages are shared.

Each partition belongs to only one consumer.

Across DIFFERENT Consumer Groups
Group A
Group B
Group C

Messages are copied logically to each group.

Every group gets all messages.

Interview Answer

Consumers within the same consumer group share the workload and 
each partition is assigned to only one consumer in that group. 
However, different consumer groups consume independently, 
so all consumer groups receive the same messages from the topic.
